# 万象 WANXIANG · 系统设计文档

> 基于 OASIS 内核改造的多智能体人群模拟预测平台（中国本土化，对标 Aaru）
>
> 版本：v1.0　日期：2026-06-13　状态：待评审

---

## 0. 一页纸总览（TL;DR）

**做什么**：一个用成千上万 AI 虚拟人模拟真实人群、为企业预测"消费/营销/舆情"反应的 SaaS 平台。用户用自然语言描述研究目标，平台调度虚拟人群推演，输出带可信度标签的预测报告。是传统问卷/焦点小组的更快、更便宜替代品。

**怎么做**：复用 OASIS 成熟的多智能体异步模拟内核，在其上重构出"平台抽象 + 场景插件 + 业务产品"三层，把"国内外平台差异"下沉为可配置的"方言层"。

**核心差异化**：Aaru 的虚拟人是孤立答题（被批"高级合成问卷"）；万象的虚拟人**会互相种草、互相影响**（L2 社交层），更接近真实社会传播——这是 OASIS 内核赋予的、Aaru 不具备的能力。

**已锁定的 11 条关键决策**：见 §1。

## 1. 已确认的关键决策（Decision Log）

| # | 决策项 | 结论 |
|---|--------|------|
| D1 | 交互形态 | 对话融合式 UI（`chat.html`）：清爽对话为主，可视化（粒子群/驾驶舱/报告）按需召唤为"工件" |

| D4 | 校准闭环 | MVP 含轻量版：模拟结果对齐公开统计分布，计算 Spearman / RMSE，输出"保真度"标签 |
| D5 | 技术路线 | OASIS 内核基座 + 平台抽象重构（方案 B+），先一次性重构出引擎骨架再插件式开发 |
| D6 | 部署 | 单机低成本：云服务器 + DeepSeek API + SQLite + 进程内队列，MVP 避开 GPU |
| D7 | 模拟范式 | 路线丙：以"媒体环境派"为主（平台溶进画像+场景），保留 OASIS 社交涌现为差异化 |

| D11 | L3 平台范围 | MVP 支持 Twitter / Reddit / 小红书 / 抖音 / 微信；微信排最后（强关系私域，最重）；后续再加其他 |

## 2. 产品定位与命名

- **产品名**：万象 WANXIANG（可替换）
- **一句话**："森罗万象，模拟万物" —— 用 AI 虚拟人群预测真实世界反应
- **目标客户**（MVP）：品牌方市场/洞察部门、广告代理公司、MCN、咨询机构
- **核心场景**：新品概念测试、口味/包装/定价测试、广告创意 A/B、营销投放预测、品牌舆情推演
- **价值主张**：6 周 → 2 小时；¥20 万 → ¥数千；且可做"假设性/未来场景"传统调研做不了的事

## 3. 整体架构（四层）

```
┌──────────────────────────────────────────────────────────────┐
│ ④ 业务/产品层 (BUSINESS, 闭源)                                  │
│   对话式 UI · API 网关(FastAPI) · 任务编排 · 造人画像服务       │
│   · 校准服务 · 报告服务 · 多租户 · 模型配置 · 计费              │
├──────────────────────────────────────────────────────────────┤
│ ③ 场景插件层 (SCENARIOS, 可插拔)                                │
│   消费洞察 · 营销预测 · 品牌舆情   [预留: 新场景注册接口]        │
├──────────────────────────────────────────────────────────────┤
│ ② 模拟编排层 (SIMULATION)                                       │
│   三层动作空间(L1/L2/L3) · 媒体环境注入 · 三档模式调度          │
│   · 群体涌现 · 反事实对照                                       │
├──────────────────────────────────────────────────────────────┤
│ ① 引擎内核 (ENGINE, 复用 OASIS + 一次性重构)                    │
│   Channel 异步解耦 · Agent 运行时 · trace 行为日志             │
│   · AgentGraph 关系图 · step 编排 · CAMEL 模型调用             │
└──────────────────────────────────────────────────────────────┘
```

**层间依赖方向**：上层依赖下层接口，下层不知道上层存在。业务层与引擎/平台层**物理隔离**（不同目录/包），保证 Apache 2.0 合规边界清晰。

## 4. 七大功能模块详细设计

### M1 数据接入与画像库

**职责**：接入构建虚拟人所需的多源数据，并做合规处理。

- **数据源**（MVP，全部走合规路线）：
  - 公开统计：国家统计局年鉴、人口普查（年龄/性别/城市/收入分布）
  - 行业数据：公开行业报告、品类消费报告
  - 合规面板：授权采购的脱敏消费面板数据
  - 社媒情感：公开聚合的品类舆情信号
  - 合成数据：填补稀疏细分群体（概率生成）
- **合规技术**（D11 中国落地关键）：
  - 差分隐私（Differential Privacy）：面板数据加噪
  - 数据脱敏：去除个人标识
  - 合成补缺：稀疏群体用生成数据，不碰真实个人敏感信息
  - 不接入任何个人可识别信息（PII）
- **接口**：`DataSource.load(segment_spec) -> DistributionProfile`
- **MVP 范围**：轻量——先支持"统计年鉴 + 1 个行业报告 + 合成补缺"，面板数据接口预留

### M2 造人引擎（Persona Builder）

**职责**：把数据分布转化为成千上万个带丰富特质的虚拟人（agent）。

- **每个虚拟人 = 200+ 维特质**，分三组：
  1. **人口标签**：年龄/性别/城市/收入/教育/职业
  2. **个性向量**：价格敏感度、尝鲜意愿、健康意识、从众倾向、品牌忠诚、风险偏好…（行为经济学+社会心理学框架）
  3. **媒体消费习惯**：常看哪些媒体/平台、信任哪类信息源（**这是 D7"平台溶进画像"的落点**）
- **构造流程**：分布抽样 → 特质赋值 → 生成 system prompt → 实例化为 engine 的 agent
- **与分布对齐**：抽样后验证人口结构与普查分布一致（连接 M5）
- **接口**：`PersonaBuilder.build(distribution, n, trait_config) -> List[Persona]`
- **复用 OASIS**：`UserInfo` → system message 机制保留并大幅扩展（从简单画像扩到 200+ 特质）

### M3 场景编辑（Scenario Composer）

**职责**：把用户的研究意图转化为可执行的模拟配置。

- **双模式输入**：
  - 自然语言（主）：用户说"测青提/白桃/海盐荔枝三口味 + 5/6/8 元定价"，AI 解析成结构化场景
  - 结构化（辅）：表单微调
- **场景四要素**：研究目标 / 投放材料（图文/视频/卖点）/ 媒体环境 / 干预变量（口味×价格等组合）
- **变量组合**：自动展开为 N 个对照场景（如 3 口味 × 3 价格 = 9 组），连接反事实模拟
- **接口**：`Scenario.compose(nl_input | struct_input) -> SimulationConfig`
- **场景插件**：消费/营销/舆情各自定义"问什么、看什么、怎么算结果"，实现 `ScenarioPlugin` 抽象

### M4 模拟内核（Simulation Core）

**职责**：调度虚拟人群在场景中决策/互动，产生可追溯结果。这是复用 OASIS 最多的模块。

- **三档模式**（D9）：
  - `decision_only`（L1）：画像 → 看材料 → 输出决策（评分/选择/点击概率）。**不启动社交循环**，快且便宜。对标 Aaru。
  - `social`（L1+L2）：启动 Channel + AgentGraph，agent 多轮互动（种草/劝退/影响）→ 态度演化 → **再决策**（D10 甲方案）。OASIS 差异化。
  - `platform`（L1+L2+L3）：在 social 基础上，按 L3 方言约束动作集与规则（关系可见性、平台特有动作）。
- **媒体环境注入**：场景材料以"信息流"形式推送给 agent（复用并改造 OASIS `recsys` 推荐流 → scenario feed）
- **三层动作空间**：见 §5
- **复用 OASIS**：Channel、AgentGraph、trace、step 编排、CAMEL 调用全部复用
- **MVP 范围**：`decision_only` + `social` 完整；`platform` 框架 + L3 方言分批（见 §5.4）

### M5 校准与保真评估（Calibration）⭐ 护城河

**职责**：证明"模拟结果可信"，给出可信度标签。这是区别于"玩具"的关键，也是 Aaru 商业说服力来源。

- **轻量版闭环**（D4，MVP）：
  1. 模拟产出群体分布（如各口味偏好占比）
  2. 与公开 ground-truth 分布对比（统计年鉴/行业报告里的已知分布）
  3. 计算三指标：**Spearman 相关**（排序一致性）、**RMSE**（平均误差）、**欧氏距离**（多维差异）
  4. 输出"校准保真度 91%"这类标签 + 置信区间
- **完整版**（V2 预留）：对抗性验证、迭代调参缩小"模拟 vs 真实"差距、强化学习进化 agent
- **接口**：`Calibrator.evaluate(sim_result, ground_truth) -> FidelityReport`
- **诚实标注**：报告中明确"结果为概率预测，建议结合业务判断"——规避 Aaru 被批"过度声称"的坑

### M6 报告生成（Reporting）

**职责**：把海量 trace 聚合成决策就绪的洞察报告。

- **报告内容**：
  - 预测结论（推荐方案 + 核心指标 + 置信区间）
  - 多维预测（口味×价格购买意愿等，带置信区间）
  - **因果归因**："为什么他们这样选"——量化各因素贡献（卖点/渠道/价格弹性）
  - **反事实对照**：A/B 方案推演（"如果加大小红书投放会怎样"）
  - 劝退原因构成、群体情绪演化
- **技术**：trace 聚合 → 统计分析 → LLM 生成自然语言解读 → 结构化为报告 + 工件卡
- **因果归因实现**：利用 OASIS 可重复 `reset` 的特性跑对照组（移除某变量再模拟，对比差异）
- **接口**：`Reporter.generate(sim_result, fidelity) -> Report`
- **输出形态**：对话内"工件卡" + 全屏报告 + PDF 导出

### M7 平台封装（Platform/SaaS）

**职责**：把引擎封装成可商用的多租户 SaaS。

- **多租户**：租户隔离（沙盒/数据/结果），每租户独立配额
- **模型配置**（D3）：租户级 LLM 配置——默认 DeepSeek，可切换。封装 OASIS `ModelFactory`
- **计费**：按模拟次数/规模/能力档位（L1/L2/L3 三档=三价位，D9）计量
- **任务编排**：模拟任务异步队列（MVP 进程内队列，V2 上 Redis/Celery）
- **API 网关**：FastAPI，对接前端与第三方集成（预留 API/SDK）
- **MVP 范围**：基础多租户 + 模型配置 + 简单计量；企业级 SSO/审计预留

## 5. 三层动作空间设计（核心创新）

### 5.1 三层定义

```
L1 决策响应 (Decision)  ── Aaru 路线, 平台无关
   评分 / 选择 / 点击概率 / 态度极性 / 愿付价格
   "看完材料你怎么选" —— 所有场景、所有平台通用, 直接产出可预测分布

L2 通用社交 (Social)    ── OASIS 内核, 跨平台共性, 抽象语义
   发布 / 转发 / 点赞 / 反馈 / 评论 / 关注 / 收藏 / 私信 / 屏蔽
   不绑定任何具体平台; agent 互相影响

L3 平台方言 (Dialect)   ── 国内外差异在此消化, 声明式映射
   把 L2 抽象动作翻译成具体平台的叫法/规则/限制/关系语义
```

### 5.2 组合规则（D9）

- 合法组合：`L1` / `L1+L2` / `L1+L2+L3`，**逐层叠加，不可跳层**
- 非法：`L1+L3`（缺 L2 抽象，方言无映射对象）、`L2 单独`（无决策输出）
- 实现为 `SimulationConfig.mode ∈ {decision_only, social, platform}`
- **能力档位 = 价位档**：decision_only（基础）/ social（专业）/ platform（企业）

### 5.3 L1+L2 语义（D10 甲方案）

社交互动**回流影响**最终决策：
```
agent 初始决策(L1) → 多轮社交互动(L2: 看到他人种草/劝退) → 态度演化 → 最终决策(L1')
预测结果 = 最终决策 L1'（经过群体影响后的）
```
这是"信号叠加"而非"功能叠加"——L2 的存在改变 L1 的输出。差异化卖点：**"Aaru 是孤立答题，万象的虚拟人会互相影响。"**

### 5.4 L3 平台方言（D11）

每个平台 = 声明式 yaml 配置 + 少量特有逻辑。示例：

```yaml
# l3_dialects/xiaohongshu.yaml
extends: social_base
relationship: weak              # 关系语义
feed_algorithm: recommend       # 信息流类型
supported_actions:
  publish:  {alias: 发笔记}
  like:     {alias: 点赞}
  collect:  {alias: 收藏, extra: {合集: true}}
  comment:  {alias: 评论, extra: {追评: true}}
  purchase: {alias: 下单}
disabled_actions: [downvote]    # 该平台无此动作
```

**平台差异矩阵**（L3 消化）：

| L2 抽象动作 | Twitter | Reddit | 小红书 | 抖音 | 微信 |
|------------|---------|--------|--------|------|------|
| 发布 | Tweet | Post | 发笔记 | 发视频 | 朋友圈/视频号 |
| 转发 | Repost/Quote | Crosspost | 转发 | 分享 | 转发(仅好友) |
| 正反馈 | Like | Upvote | 点赞 | 点赞 | 点赞 |
| 负反馈 | — | Downvote | — | 不感兴趣 | — |
| 收藏 | Bookmark | Save | 收藏+合集 | 收藏 | 收藏 |
| 深度消费 | — | — | 看评论区 | 完播 | 读完 |
| 转化 | — | — | 下单 | 下单 | 小程序下单 |
| 关系语义 | 弱(关注) | 无(社区) | 弱 | 弱(算法) | **强(好友)** |

**实现批次**（按难度排序，避免最难的卡住进度）：
- M0 重构时：**Twitter + Reddit**（OASIS 已有，搬运进 yaml 范式，近乎免费）
- L3 第一批：**小红书 + 抖音**（弱关系/算法流，与现有模型接近）
- L3 第二批：**微信**（强关系私域，需 engine 支持"关系可见性过滤"，牵动底层，排最后）

> ⚠️ 微信的"仅好友可见""群聊传播路径"要求 L2/engine 支持基于 AgentGraph 的信息可见性过滤，这是 5 个平台里唯一会反向触及内核的，故单列第二批。

## 6. 对话式 UI 集成（D1）

`chat.html` 的"AI 首席模拟官"是整个系统的统一入口：

```
用户自然语言
   ↓ (app/chat: NL 解析)
SimulationConfig (场景+人群+模式+模型)
   ↓ (M3 场景 → M2 造人 → M4 模拟 → M5 校准 → M6 报告)
结果流式返回
   ↓
对话内呈现:
  · 流式状态卡 (模拟进度, 缓解延迟焦虑)
  · 工件卡 (报告摘要, 可"展开全屏")
  · 追问气泡 (引导下一步)
右侧数据面板: 迷你粒子群 + 实时指标 (旁观)
全屏覆盖层: 驾驶舱 / 完整报告 (高光时刻, 按需召唤)
```

- 左栏"沙盒"= 每个人群数字孪生（天然实现 Reset：每沙盒独立，可重置）
- 三档模式对用户透明：用户说需求，系统按场景自动选 mode（或用户在设置里指定档位）

## 7. 部署与成本（D6）

### 7.1 MVP 单机架构

```
一台 2-4 核 8G 云服务器 (阿里云/腾讯云轻量, ~¥100/月)
└─ 一个 Python 进程:
   ├─ FastAPI (API + 托管 chat.html 静态资源)
   ├─ import engine/ (OASIS 内核作为库)
   ├─ 进程内任务队列 (asyncio, 不上 Redis)
   └─ SQLite (沙盒/用户/结果, 单文件)
LLM: DeepSeek API (按量, 不自建 GPU)
推荐流: OASIS 自带 reddit 热度/random 算法 (纯 CPU, 不加载 BERT)
```

- **关键省钱点**：避开 GPU。LLM 走云 API；推荐系统 MVP 用轻量算法（不用 twhin-bert）
- **月成本**：服务器 ~¥100 + DeepSeek 按量（单次 5 万 agent 模拟约几元~几十元）

### 7.2 演进路径

| 阶段 | 触发 | 升级 |
|------|------|------|
| 阶段0 MVP | — | 单机 + SQLite + 进程队列 + DeepSeek |
| 阶段1 有付费客户 | 并发上升 | 8核16G + PostgreSQL + Redis/Celery（仍可同机或云托管按量） |
| 阶段2 规模化 | 冲百万级 | 多机 + GPU + 本地模型；engine 可从"基于 OASIS"替换为自有引擎（上层业务不动） |

## 8. 中国合规方案

| 风险点 | 措施 |
|--------|------|
| 个人信息保护（PIPL） | 不接入 PII；面板数据走授权采购 + 脱敏 + 差分隐私；稀疏群体用合成数据 |
| 数据安全法 | 数据本地化存储；模型可选本地部署（D3 客户偏好） |
| 舆情场景敏感性 | 品牌舆情场景限定"商业品牌"，不碰公共/政治议题；内容审核接口预留 |
| 结果误导风险 | 报告强制标注"概率预测，建议结合业务判断"，给置信区间，不做绝对承诺 |
| 开源合规（Apache 2.0） | engine/ 保留 OASIS 版权头、标注修改；业务层物理隔离闭源 |

## 9. 实现里程碑（M1-M7 全范围，分阶段交付）

> 用户目标：M1-M7 全部实现。以下为达成该目标的分阶段路径（避免一次性失控）。

```
M0  引擎抽象重构 (地基, 必须最先)
    · Fork OASIS → 切分 engine/ + platform_base.py 接口
    · Twitter/Reddit 逻辑搬入 L3 yaml 范式 (验证不搬坏)
    · 产出: 干净的引擎骨架 + 动作三层框架

M1  造人 + 模拟主链 (decision_only)
    · M1数据接入(轻) + M2造人(200+特质) + M4 decision_only
    · 跑通: 分布 → 造人 → 看材料 → 输出决策分布

M2  社交涌现 + 校准 + 报告
    · M4 social档(L1+L2, 甲方案) + M5轻量校准 + M6报告(含因果/反事实)

M3  产品化
    · M7多租户+模型配置+计费 + 对话UI接入(app/chat) + 单机部署

M4  场景 + 平台方言铺开
    · M3三场景插件(消费/营销/舆情)
    · L3第一批(小红书+抖音) → L3第二批(微信)
```

每个里程碑结束 = 一个可演示、可验证的增量。后续每个里程碑会单独出实现计划（plan）再编码。

## 10. 关键风险与应对

| 风险 | 应对 |
|------|------|
| M0 重构成本被低估（platform.py 1642行要拆） | 单列为独立里程碑，先于一切功能；只搬不加料 |
| 微信强关系模型牵动内核 | 排 L3 最后；先确认 L2/engine 关系可见性扩展点 |
| "套壳问卷"质疑（Aaru 同款批评） | 用 L2 社交涌现（D10甲）+ M5 校准双重背书 |
| 校准 ground-truth 数据不足 | MVP 先用公开统计年鉴/行业报告；面板数据逐步补 |
| LLM 成本随规模膨胀 | decision_only 档省钱；抽样渲染；DeepSeek 默认；批处理 |
| 大规模性能（OASIS SQLite 单平台协程瓶颈） | MVP 规模（5-10万）够用；规模化时走阶段2自有引擎 |
